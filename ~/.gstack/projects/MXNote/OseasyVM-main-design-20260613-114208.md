## Problem Statement

中国学术研究者（硕博研究生、博士后）在日常研究中面临严重的跨设备笔记碎片化问题。他们当前拼凑多个工具——手机备忘录、桌面Typora/VS Code、云盘手动同步——每一次设备切换都打断思考流。市场上Obsidian、Notion、语雀等工具在移动端体验差、同步不可靠（冲突/覆盖/延迟），且无一支持HarmonyOS原生体验。

核心痛点不是"没有笔记工具"，而是**没有一个工具能让研究者在任何设备上无缝继续思考**。

## Demand Evidence

- 用户自身是学术研究者，每天经历跨设备笔记切换的痛苦
- 具体场景：手机上记录灵感 → 电脑上无法直接继续 → 思考被打断 → 大量碎片化笔记无法整理
- 当前工作流：手机备忘录 + Typora + 微信文件传输 = 高摩擦、易丢失、无结构化
- Obsidian Sync ($4-8/月) 存在已知的同步覆盖/冲突bug，用户论坛大量投诉
- HarmonyOS生态内无Markdown原生笔记工具——市场真空

## Status Quo

目标用户当前的工作流：
- **桌面端**：Typora/VS Code写Markdown，Obsidian管理知识库，Zotero管理文献
- **移动端**：手机备忘录快速记录，微信发给自己暂存
- **同步**：云盘（百度网盘/OneDrive）手动传文件，或Git管理.md文件
- **代价**：每天至少浪费15-30分钟在格式转换、文件同步、找回丢失内容上。更严重的是**思维中断**——灵感到来时无法立即在正确位置继续写作。

## Target User & Narrowest Wedge

**核心用户**：中国理工科硕博研究生（25-35岁）
- 主力设备：HarmonyOS手机 + 华为MatePad平板 + Windows/Mac笔记本
- 工作流：大量阅读PDF文献 → 做标注和笔记 → 写论文/实验报告
- 痛点时刻：在手机上读到好文献想记笔记 → 切换到桌面时笔记无法无缝衔接

**最小切入**：**跨设备接力编辑**（Continuity Editing）
- 在一个设备上打开笔记并输入 → 拿起另一设备 → 收到通知"检测到活跃笔记" → 点击确认 → 接管编辑
- 同步粒度：`{content, cursorOffset, scrollY}` —— 不只是文件，而是**思考会话**
- 利用HarmonyOS `distributedDataObject` API实现OS级状态同步，局域网零延迟
- V1撤销栈为本地操作（分布式Undo是已知难题，留到V2+调研OT/CRDT方案）
- **交互模式**: 远端更新→通知提示→用户确认接管（非静默controlled TextArea，避免IME/光标破坏风险）

## Constraints

- HarmonyOS ArkTS/ArkUI技术栈（DevEco Studio）
- 初始团队规模小（单人/小团队）
- 需要同时支持手机、平板、2in1三种鸿蒙设备形态
- 必须兼容华为小艺AI助手接口
- 目标市场为中国大陆，优先中文体验
- Windows/Mac用户暂时无法覆盖（鸿蒙优先策略）

## Premises

1. ✅ **研究者存在跨设备Markdown笔记的真实痛点**，现有方案无法满足 — 用户确认
2. ✅ **HarmonyOS分布式能力是真正的技术壁垒** —— 用户押注华为全家桶在国内研究者的渗透率增长 — 用户坚持（对抗了第二意见的挑战）
3. ✅ **LLM深度集成（小艺+自定义模型）是核心差异化价值** —— 用户确认

## Cross-Model Perspective

独立AI第二意见的核心输出：

**Steelman**: MXNote不是Markdown编辑器，而是"持久化思考面"——设备是通向同一连续认知会话的可丢弃视口，LLM是拥有所有笔记全文语境的"环境式研究副驾驶"。

**关键洞察**: "跨设备切换打断了我的思考流"——每个竞争对手以`.md`文件为保护单元，MXNote应该以**思考会话**为保护单元。MVP必须同步光标位置、输入中的字符、滚动位置和撤销历史。

**挑战的前提（用户拒绝）**: 鸿蒙分布式壁垒需要目标用户实际拥有多个鸿蒙设备。如果中国研究生主力写作设备是Windows笔记本，壁垒变成牢笼。用户基于自己的观察和趋势判断维持了原前提。

**原型建议**: 48小时两端接力编辑Demo——`TextArea` → `distributedDataObject` → 另一端`TextArea`。零后端。只做纯文本+光标同步。验证核心假设。

## Approaches Considered

### Approach A: 接力编辑验证器（S/低风险）
- 48小时技术验证，纯鸿蒙分布式总线
- 不做Markdown渲染、不做LLM、不做文件管理
- 快速验证核心假设，失败沉没成本极低

### Approach B: 全栈研究平台（XL/高风险）
- 一次性完整架构：分布式引擎+Markdown渲染+LLM插件体系+云同步
- 3-6个月开发周期，直接产出可发布产品
- 验证成本高，核心假设有误则损失大

### Approach C: 渐进式架构（M/中风险）— **[已选]**
- V1: 接力编辑技术验证 → V2: Markdown编辑器+基础同步 → V3: LLM集成
- 同步层设计为"分布式总线+轻量云API"双通道，预留Web扩展口
- 务实路线：利用鸿蒙优势但不锁死，可渐进交付

## Recommended Approach

**方案C: 渐进式架构**，分三阶段交付：

```
V1 (2-4周): 接力编辑MVP
├── 分布式数据对象同步引擎 (distributedDataObject)
├── 单笔记纯文本编辑+光标/滚动同步
├── 手机↔平板接力Demo
└── 技术验证完成 → 进入V2

V2 (10-16周): Markdown笔记核心
├── Markdown编辑器: CommonMark + GFM表格 + LaTeX数学公式(最小可用子集) + 脚注
│   └── KaTeX移植技术说明: ArkTS无DOM，KaTeX(~20K行JS)不可直接移植。
│       策略: V2用WebView渲染数学公式作为降级方案; 原生渲染仅实现最小LaTeX子集
│       (行内公式、分数、上下标、希腊字母)。完整KaTeX等价渲染为V3+
├── 编辑/预览切换模式 (单窗格切换，非实时分栏——分栏预览推迟到V3)
├── 语法高亮: 关键字匹配型最小实现 (完整highlight.js移植为V3+)
├── 多笔记管理 (创建/删除/重命名/标签系统——文件夹为V3)
├── 全文搜索 (SQLite FTS5)
├── 图片粘贴+本地存储 (5MB以下跟随笔记同步，以上仅为本地)
├── 笔记导出: 单篇.md导出，批量.zip导出
├── 本地持久化 (SQLite存储，AES-256单设备加密，含V1→V2数据迁移)
├── 跨设备同步完善 (LAN优先+云兜底，含离线队列)
├── LWW冲突策略 + 时钟偏差保护 (30秒偏差窗口内视为真正冲突，保留双版本)
├── 撤销/重做: 本地操作; 分布式撤销为V3调研项
├── 基础设置页面 + 深色模式支持
├── TalkBack无障碍标注 (所有交互元素)
├── 错误处理: 网络/存储/解析三级错误分类 + 用户可见错误状态
└── 内部测试版本 → 进入V3

V3 (6-10周): LLM集成+云端+发布准备
├── LLM Provider: 小艺 (生态要求) + OpenAI兼容接口 (覆盖DeepSeek/Ollama)
│   └── 小艺应急预案: 若V3启动时小艺API不支持自定义Prompt注入，
│       切换为DeepSeek为主力模型 (中文市场性价比优势)，小艺集成顺延至后续版本
├── 实时流式LLM响应 (按Token渲染)
├── 上下文感知——选中文本 → AI解释/扩展/提问
├── 笔记全文作为LLM上下文 (用户显式授权)
├── 实时分栏预览 (编辑/预览左右分屏)
├── 云端同步服务上线 (TLS传输 + 服务端加密)
│   └── 端到端加密(E2EE)推迟至V4——V3先交付传输层安全
├── 应用商店上架准备
└── Beta公测 (50人)
```

### 核心组件规格

**SessionEngine（会话状态管理）**

会话是 MXNote 的原子同步单元。一次会话 = 用户打开一个笔记到切换离开。

```
interface Session {
  noteId: string
  content: string          // 笔记全文
  cursorOffset: number     // 光标位置 (UTF-16 code unit offset)
  scrollY: number          // 滚动位置
  lastModified: number     // Unix timestamp, LWW冲突裁决依据
  deviceId: string         // 最后写入设备
}

// 生命周期:
// begin(noteId) → 从LocalStore加载 → 订阅DistributedObject
// onContentChange() → 写入DistributedObject → 触发远端同步
// onRemoteChange() → 检测冲突(timestamp) → LWW裁决 → 更新本地
// end() → 持久化到LocalStore → 释放分布式订阅
```

**MarkdownParser（解析器规格）**

- 标准: CommonMark 0.30 + GFM扩展（表格、任务列表、删除线）
- 数学: LaTeX行内公式 `$...$` + 块级公式 `$$...$$`，渲染引擎移植KaTeX
- 其他: 脚注 `[^1]`、图片 `![](url)`、代码块语法高亮（highlight.js移植）
- 架构: 增量解析（只重新解析变更段落），AST → ArkUI组件树映射

**LLMProvider（插件接口）**

```typescript
interface LLMProvider {
  readonly id: string           // 'xiaoyi' | 'openai-compat'
  readonly displayName: string
  readonly requiresAuth: boolean

  chat(request: ChatRequest): Promise<ChatResponse>
  chatStream(request: ChatRequest): AsyncIterable<ChatChunk>  // 流式响应
  cancel(): void
}

interface ChatRequest {
  messages: { role: 'system'|'user'|'assistant', content: string }[]
  contextNoteIds?: string[]  // 附加上下文笔记ID
  selectedText?: string       // 用户选中的文本
  maxTokens?: number
}

interface ChatChunk {
  delta: string         // 增量文本
  finishReason?: 'stop' | 'length' | 'error'
}
```

V3首批上线: 小艺 + OpenAI兼容接口(一个适配器覆盖DeepSeek + Ollama)。Provider接口已设计为后续添加任意模型无需重构。

**SyncManager（同步协调）**

同步通道选择策略：
- 设备在同一局域网 → 优先 DistributedObject (近场零延迟)
- 设备跨网络/分布式总线不可用 → 降级 CloudSync (WebSocket)
- 完全离线 → 本地操作入队，恢复连接后按时间戳合并（LWW）
- 冲突检测：同一noteId + 并发修改 → `lastModified`较高者胜出
- 时钟偏差保护: 两次编辑的时间差在30秒偏差窗口内 → 视为真正并发冲突 → 保留双版本，用户手动裁决
- 被覆盖版本保留为可见的冲突条目（笔记列表内显示冲突徽章+时间戳），点击打开并排对比视图

## Engineering Review Decisions (2026-06-13)

经过 `/plan-eng-review` 完整审查（5架构+3代码质量+1测试+1性能+1关键缺口+交叉模型外部声音）。

### 架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| Session对象结构 | **强制扁平** | distributedDataObject不同步嵌套属性，架构门禁防止退化 |
| 超大笔记策略 | **分块+原子提交** | chunk_count协议，所有块到齐后一次性组装渲染 |
| 同步节流 | **内容立即，光标150ms去抖** | 内容不能丢，光标/滚动可接受短暂延迟 |
| 会话发现 | **自动检测+通知确认** | 平板靠近时检测活跃Session，弹通知"继续编辑？"，用户确认后接管 |
| Markdown解析 | **源文本同步+本地各自解析** | 不耦合同步层和解析层，每个设备独立渲染 |
| 交互模式 | **通知式接管** | 远端更新→用户手动确认→接管焦点（避免controlled TextArea的IME/光标风险） |
| 性能优化 | **远端内容300ms解析节流** | 减少远端频繁推送导致的重解析卡顿 |

### 代码质量决策

| 决策 | 选择 |
|------|------|
| 词法层复用 | Parser内嵌高亮Token产出，避免Lexer重复 |
| 错误处理 | Result<T,E>模式（Rust风格），强制调用方处理 |
| 分块协议 | chunk_count+chunk_index，原子组装，30s超时重请求缺失块 |

### 测试策略

- 单元测试：所有可单设备测试路径全覆盖（MarkdownParser AST正确性、SessionEngine状态机、LWW裁决）
- 手动矩阵：9格（3设备组合 × 3网络状态）
- 钉子测试先行：V1启动前必须先验证 distributedDataObject 在目标网络的实际传输性能

## Open Questions

1. **设备生态调研**: 目标用户（中国理工科硕博）实际设备组合是什么？华为全家桶比例？需要50-100人问卷验证
2. **小艺接口能力**: 小艺AI的API是否支持自定义Prompt和上下文注入？需要确认技术可行性
3. **distributedDataObject性能与上限**: 大文本（10KB+）同步延迟？有效载荷上限是多少？学术笔记（文献综述、论文草稿）可达50-200KB。若API有64KB限制或100KB以上性能骤降，V2必须实现分块同步或完全降级到云同步
4. **商业模式**: 免费+付费功能分层？还是纯订阅制？定价策略待定
5. **协作编辑**: 多人实时协作是V4还是独立产品？

## Data Migration Strategy

| 版本跨越 | 迁移内容 | 迁移方式 |
|---------|---------|---------|
| V1→V2 | 纯文本笔记→SQLite结构化存储 | V2首次启动自动扫描distributedDataObject残留数据，导入SQLite，标记迁移完成 |
| V2→V3 | 新增LLM对话记录表、API Key加密存储 | SQLite schema升级（新增表），无需数据迁移，向下兼容读取V2数据 |

**原则**: 迁移是单次的、自动的、不可见的。用户永远不需要手动导出导入。

## Security & Privacy

研究笔记可能包含未发表数据、实验原始结果和敏感发现。安全不是附加功能。

| 层级 | 策略 | 实现 |
|------|------|------|
| 本地存储 | AES-256加密 | SQLite文件级加密，密钥派生自设备锁屏密码/HarmonyOS生物认证 |
| 近场传输 | 鸿蒙分布式总线加密 | HarmonyOS系统级安全通道，不需要应用层额外处理 |
| 云端传输 | TLS 1.3 | WebSocket over TLS，证书固定 |
| LLM上下文 | 用户显式授权 | 发送笔记全文到LLM之前弹窗确认，可选"仅发送选中文本" |
| 云端存储 | TLS传输 + 服务端加密(静置) | V3实现传输层+存储层安全; 端到端加密(E2EE)推迟至V4 (跨设备密钥管理复杂性高，V3先交付传输层安全) |

**隐私承诺**: MXNote不在服务器端存储可读的笔记内容。V1完全离线无需服务器。V2 LAN阶段：设备各自用本地密钥加密SQLite，分布式总线传输使用鸿蒙系统加密通道无需应用层额外处理。V3云端阶段：TLS传输+服务端加密存储。E2EE（端到端加密，服务端完全不可读）为V4路线图项。

**跨设备加密说明**: V1/V2各设备使用各自锁屏密码派生本地加密密钥。跨设备同步依赖鸿蒙分布式总线的系统级安全通道(传输层加密)，应用层不额外处理。V4 E2EE将实现基于华为账号的全设备共享主密钥。

## Offline Behavior

MXNote是**本地优先**架构，离线不是降级模式而是默认模式。

- **离线创建/编辑**: 始终可用。所有操作先写入LocalStore
- **重连同步**: 设备恢复网络后，SyncManager自动合并离线期间的修改队列
- **LAN降级**: 分布式总线不可用 → 尝试云同步 → 都不可用 → 本地操作，标记 `pendingSync`
- **冲突: LWW(时间戳)裁决 + 冲突副本**: 离线时多设备修改同一笔记 → 较新修改胜出，被覆盖版本保留为 `{noteId}.conflict.{timestamp}.md`。用户永远不会静默丢失数据
- **离线指示器**: UI显示当前同步状态（绿色=实时同步中，黄色=离线排队，红色=同步失败）

## Testing Approach

分布式系统三端多状态组合，测试策略分三层：

| 测试类型 | 范围 | 工具 |
|---------|------|------|
| 单元测试 | MarkdownParser (AST正确性)、SessionEngine (状态机)、SQLite CRUD | ArkTS测试框架 (ohosTest) |
| 集成测试 | SyncManager同步逻辑、LWW冲突裁决、离线队列重放 | 多设备自动化测试 |
| 手动验证矩阵 | 手机+平板+2in1三端组合 × (在线/LAN离线/完全离线)三种网络状态 | 手动测试清单 |

V1重点：多设备手动验证分布式同步延迟和稳定性。
V2重点：Markdown渲染兼容性 + LWW冲突场景覆盖。
V3重点：LLM Provider切换 + 云端同步端到端测试。
